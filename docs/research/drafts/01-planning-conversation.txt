Let me work through each one and then synthesize how the pipeline changes.

On Point 1 — the Learner Knowledge Graph
This is the piece that was conspicuously missing from the architecture, and your framing of it as a graph overlay is exactly right. The current sketch treats every learner as starting from zero, which is almost never true. A person who wants to learn Mandarin but already speaks Japanese has a radically different optimal curriculum than someone with no tonal language experience. A programmer learning data science already has half the dependency graph filled in. The system was doing skill decomposition but not learner decomposition, which means the Curriculum Architect was designing from-scratch curricula when it should be designing gap-filling curricula.
The two-pass structure you're describing makes strong sense because the information available changes between passes. In the first pass, before deep research has been done, you don't yet have the skill's dependency graph to map against. But you can gather broad background: what related skills does the learner have, what's their learning history in adjacent domains, what frameworks or mental models do they already carry? This is high-signal information that should steer the Skill Researcher — it tells the researcher which transfer pathways to investigate, which analogies to look for, and which parts of the skill landscape to explore more deeply because they might connect to what the learner already knows.
In the second pass, after the dependency graph exists, you can do something much more precise: walk through the graph's nodes and assess the learner against each one. "Do you know what a chord progression is? Can you explain what a V-I cadence resolves to? Have you ever played a stringed instrument?" This produces the overlay you're describing — the learner's knowledge graph superimposed on the skill's dependency graph, with each node colored as "already have," "partially developed," or "new territory." The delta between the two graphs is the curriculum, weighted by importance and transfer potential.
The optional research/calibration loops are important too. Sometimes Pass 2 reveals that the learner has unexpected strengths or gaps that change the research priorities. Maybe the learner already has strong music theory but zero physical instrument experience — that reshapes which parts of "learning guitar" need deeper investigation. The system should be able to bounce between research and calibration until both the skill graph and the learner overlay are stable enough to design a good curriculum.
Your knowledge graph lens concept also has a beautiful motivational property: showing someone "you already have 40% of the nodes filled in" is much more energizing than presenting a blank curriculum and saying "here's everything you need to learn." It reframes the challenge from "learn this whole skill" to "bridge these specific gaps" — which is more accurate and more motivating.

On Point 2 — the "how can we" mindset
You're right to flag this. The research document's cautionary tone about wicked environments and expertise ceilings is valuable as design input — it tells us where the system needs to work harder — but it shouldn't cascade into a defensive or defeatist user experience. The user didn't ask for a lecture on the limitations of meta-learning; they asked for a learning plan.
The correction I'd make to the architecture is to reframe the Domain Assessor's output. Instead of producing a "framework fit score" with warnings, it should produce an approach strategy with a constructive path forward for every case. If someone wants expertise in negotiation by tomorrow, the system shouldn't say "that's unrealistic, here's why." It should say "here's what we can accomplish by tomorrow — a strong conceptual foundation and two practiced techniques — and here's the 12-week roadmap that builds on tomorrow's work toward the expertise level you're after." The short-term plan and the long-term roadmap are both real, actionable outputs. The calibration happens naturally through the specificity of what each plan contains, not through explicit "warning" language.
This also means the failure modes from the research (premature optimization, illusion of competence, skill decay, decomposition fallacy, expertise ceiling) should be encoded as design constraints that the system works around, not as warnings it presents to the user. The system accounts for these internally — it builds in overlearning budgets, delayed assessments, and honest progress metrics — without narrating the risk factors. The learner experiences a well-designed program; they don't need to hear about the cognitive science of why the program is designed that way unless they ask.

On Point 3 — the dashboard artifact
This makes the system significantly more tangible. A Claude Project-based dashboard (a React artifact that consumes the plugin's own structured outputs) gives the learner a persistent visual home base. I'd envision it displaying the knowledge graph overlay (skill dependency graph with the learner's current state colored in), the curriculum roadmap (where they are, what's next, projected timeline), key metrics (delayed retention scores, components mastered, practice consistency), and upcoming session content. This artifact gets regenerated or updated as the Training Conductor records new progress.
Architecturally, this means the Material Forge needs a sub-capability for dashboard generation — or the dashboard is a standalone skill that reads the system's structured data outputs and renders them. The latter is probably cleaner since the dashboard needs to be regenerated frequently as progress data changes.

On Point 4 — bidirectional data flow
This closes a loop that was open in the current sketch. Right now the system exports data (Anki decks, progress JSON, curriculum markdown) but never imports anything back. If the learner is doing Anki reviews outside of Claude sessions, that review data contains incredibly valuable information about retention, difficulty, and learning velocity for specific components. Similarly, if the learner is taking notes in Obsidian or practicing in a real-world context and recording results, that data should flow back in.
This means the knowledge graph representation needs to be a living document that accepts updates from multiple sources: the Training Conductor (conversational assessment data), Anki (review history and retention statistics), user self-reports (real-world practice outcomes), and any other structured input. The import format should be simple and well-documented — probably JSON with a clear schema — so that any tool can write to it.
For the Training Conductor specifically, being able to see Anki review data before a session would be powerful. It could open a session with "I see you've been reviewing the chord progression cards — your retention on major-to-minor transitions is strong but the modal interchange cards are getting flagged as hard. Let's work on that today." That level of integration makes the conversational teaching much more responsive.

On Point 5 — the 150k token session scope
This is a critical practical constraint that shapes session design in specific ways. At 150k tokens, you have room for a substantial system prompt (the Training Conductor's instructions, probably 5-10k tokens), the relevant curriculum context (current task class, recent progress, today's material — maybe 10-20k tokens of loaded context), and then roughly 120k tokens of actual conversational exchange. That's generous for a single teaching session, but it means the system can't carry the entire curriculum history, all assessment results, and the full skill dossier in every session.
The implication is that the Training Conductor needs a session loading protocol — at the start of each session, it reads the current state (from a progress file or the dashboard data), loads only the context relevant to today's session, and works within that scope. The full history lives in the project files; the session window contains only what's needed for effective teaching right now. This also means session boundaries need to be designed intentionally — a session should have a natural arc (warm-up → practice → close) that completes within the token budget, rather than sprawling indefinitely.
A related design consideration: if a topic requires more depth than one session can handle, the Conductor should be able to gracefully close a session at a natural stopping point and set up the next session's agenda, rather than running into the context ceiling mid-explanation.

How the pipeline changes
Here's how I'd revise the architecture to incorporate all five points:
The biggest structural change is that the pipeline is no longer purely linear. It gains a calibration loop between research and curriculum design, and a feedback loop from external tools and training sessions back into the knowledge graph. The knowledge graph becomes the system's central persistent artifact — it's created early, refined through calibration, and continuously updated through training and external data imports.
The revised component list looks like this. Component 1 (Domain Assessor + Initial Learner Profile) now does two things in its single pass: it classifies the skill and gathers the learner's broad background. The output is a skill classification plus a preliminary learner profile that informs the research phase. The framing is constructive throughout — "here's what we can accomplish in your timeframe, and here's the longer path if you want to go further."
Component 2 (Skill Researcher) is largely unchanged but now receives the learner profile as input, which means it actively looks for transfer pathways — connections between what the learner already knows and what the skill requires. Its dependency graph output is annotated with potential transfer points.
Component 3 is new: the Learner Calibrator. This is the focused second pass. It takes the dependency graph from the Researcher and walks through it with the learner, assessing their current state against each major node. It produces the knowledge graph overlay — the precise map of what the learner already has, what's partially developed, and what's genuinely new. It can trigger additional research if the calibration reveals unexpected gaps or strengths. The output is the annotated knowledge graph that the Curriculum Architect uses for gap-based design.
Component 4 (Curriculum Architect) now works from the gap analysis rather than from scratch. Its Selection engine prioritizes the highest-impact unfilled nodes. Its Sequencing engine can leverage existing knowledge as scaffolding for new material, and its scheduling engine accounts for the reduced scope (you don't need to schedule practice for things the learner already knows). The curriculum is a bridge from the learner's current graph to their target state.
Component 5 (Material Forge) gains the dashboard as a sub-capability. It generates all the same material types as before, plus a React dashboard artifact that visualizes the knowledge graph, curriculum progress, and key metrics. The dashboard is designed to be regenerated as progress data changes.
Component 6 (Training Conductor) gains three new capabilities: it operates within a 150k token session budget with an explicit session loading protocol, it can import external tool data (Anki review stats, user self-reports) to update the knowledge graph before each session, and it feeds assessment results back into the knowledge graph after each session. The knowledge graph evolves continuously — it's not a static snapshot but a living representation of the learner's developing competence.
The Dashboard becomes a first-class system artifact, not just a nice-to-have. It's the learner's visual home base, and it's also the primary way the system communicates the knowledge graph state — showing what's mastered, what's in progress, what's next, and how it all connects.
The data import/export layer is a cross-cutting concern. The system needs a well-defined JSON schema for the knowledge graph that external tools can both read from (to see what the system knows) and write to (to contribute new data). Anki review history, Obsidian notes, self-reported practice outcomes, and any other structured input should be consumable through a consistent import format.
