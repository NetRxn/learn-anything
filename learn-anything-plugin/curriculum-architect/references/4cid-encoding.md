# 4C/ID, Elaboration Theory, and Productive Failure: Curriculum Generation Rules

These frameworks are translated into step-by-step procedural rules for curriculum generation. Follow these rules as behavioral instructions — do not derive curriculum decisions from abstract theory.

## Elaboration Theory: The Epitome

The first lesson must be a **simplified but complete** version of the entire skill — not just the easiest piece.

**Rules for epitome design:**
1. The epitome must be a real, authentic task (not an artificial exercise)
2. It must exercise the core loop of the skill (even if simplified)
3. The learner must produce a meaningful output by the end of lesson one
4. If the learner has existing knowledge (from the knowledge graph), use it to make the epitome more sophisticated than a true beginner's version

**Examples by domain:**
- Programming: Modify a working calculator app (not "learn what a variable is")
- Guitar: Play a simplified 2-chord version of a real song (not "learn to hold the guitar")
- Cooking: Make a complete simple dish from scratch (not "learn knife cuts")
- Language: Have a formulaic real conversation (greeting, question, response) (not "learn the alphabet")
- Negotiation: Run through a simplified role-play scenario (not "learn what BATNA means")

## 4C/ID: Task Class Design

Organize the curriculum into **3-5 task classes** ordered from simple to complex.

**Rules for task class construction:**
1. Each task class represents a level of complexity for the whole skill (not a separate topic)
2. All tasks within a class have **equal complexity** but **decreasing support**
3. Support sequence within each class: worked example -> completion problem -> guided practice -> conventional problem -> independent performance
4. Support **resets** when a new task class begins (the sawtooth pattern)
5. Ensure **variability**: surface features change across tasks within a class, but deep structure stays the same

**Rules for distinguishing recurrent vs. non-recurrent sub-skills:**
- **Recurrent** sub-skills (same procedure every time): Schedule dedicated **part-task drill** for automation. Examples: typing, scales, basic arithmetic, syntax.
- **Non-recurrent** sub-skills (require judgment, vary by context): Practice only through **whole-task** variation. Never isolate into drills. Examples: design decisions, musical interpretation, writing style, diagnostic reasoning.

**Support fading schedule:**
- Task 1: Full worked example (learner observes and explains)
- Task 2: Completion problem (some steps done, learner fills gaps — fade from the end backward)
- Task 3: Guided practice (hints available on request, not proactively given)
- Task 4: Conventional problem (no support, but within familiar structure)
- Task 5: Independent problem (novel surface features, same deep structure)

**Backward fading specifically:** Remove the LAST step of a solution first, then progressively earlier steps. Each faded step includes a self-explanation prompt ("Why does this step work?").

Fading pace:
- Novice learner: 1 step per 2 problems (slow)
- Intermediate: 1 step per problem (standard)
- Advanced: 2 steps per problem (fast)

## Productive Failure: When and How

Insert productive failure moments ONLY for conceptual building blocks where:
1. Common naive theories exist that interfere with learning
2. The concept has multiple valid solution approaches
3. The learner is NOT already frustrated or struggling

**Rules for productive failure insertion:**
- Place at the **transition between task classes** (not within a class)
- The challenge problem must come BEFORE instruction
- During the struggle phase: NO scaffolding, NO correction, NO hints toward canonical solution
- Only "pumps" allowed during struggle: "What have you tried so far?" "What other approaches could work?"
- After the struggle phase: consolidation instruction that explicitly bridges the learner's attempts to the canonical solution
- Follow up with transfer: apply the concept in a new context

**When NOT to use productive failure:**
- Purely procedural components (there's no concept to struggle with)
- When the learner is below ZPD for the prerequisites (struggle won't be productive)
- When the learner shows signs of frustration or disengagement

## Mastery Gates

Place a mastery gate assessment at each task class boundary.

**Mastery gate rules:**
1. Minimum 3 assessment items (5 preferred)
2. Items at or one Bloom level above the taught level
3. Vary surface features from instruction (to test transfer, not memory)
4. Three-criteria format: Cold Recall -> Application Under Novelty -> Explain-to-Teach
5. Pass all three -> advance to next task class
6. Fail -> targeted re-teach + retest with new items (max 3 attempts)
7. After passing a gate, schedule at least one delayed retention check (1-3 days later)

## Interleaving

After the initial blocked introduction of each component:

**Interleaving rules:**
1. Practice sets should be 25% current topic, 75% review
2. Within review, weight recent topics more heavily
3. No two consecutive problems should use the same strategy
4. Include at least one pair of problems with similar surface features but different deep structure
5. Pre-frame the difficulty: "This will feel harder than practicing one thing at a time. That's intentional — it produces stronger long-term retention."

## Spacing

Session spacing should follow the Cepeda et al. research:
- **Optimal spacing gap is approximately 10-20% of the desired retention interval**
- For 1-year retention target: review every 3-5 weeks
- For 1-month retention target: review every 3-6 days
- For 1-week retention target: daily review

**Overlearning budget:**
If the skill requires durability, schedule 20+ minutes of practice beyond mastery for hyper-stabilization (Shibata et al., 2017).
